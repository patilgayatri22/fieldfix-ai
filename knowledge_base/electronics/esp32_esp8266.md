# ESP32 / ESP8266 — Repair Guide

## Common Symptoms
- Repeated brownout reset / brownout detector triggered
- WiFi connects then drops repeatedly
- Flash error or "Failed to connect to ESP32"
- Device resets every few minutes
- WiFi scan returns no networks
- Upload fails (ESP8266: port not detected)
- Serial output is garbled or missing
- Device works on USB but crashes on battery

## Likely Causes
- Brownout reset → insufficient supply current (ESP32 needs up to 500mA peak), poor USB cable, low battery
- WiFi drops → interference, signal strength marginal, DHCP lease timeout, heap fragmentation after long run
- Flash error on upload → wrong board settings, EN/RST pin not cycling, GPIO0 not pulled LOW during upload
- Periodic resets → watchdog not being fed in loop, software crash, heap out of memory
- No WiFi scan → antenna disconnected (IPEX models), wrong country code set, RF section damaged
- Upload fails → USB-serial driver missing (CP2102, CH340), wrong COM port, board not in flash mode
- Garbled serial → wrong baud rate (ESP8266 boot log = 74880 baud, application usually 115200)
- Crashes on battery → battery can't supply peak WiFi transmit current (~350mA), add capacitors

## Safe First Checks
- Check USB power supply — use 5V/1A minimum, not PC USB hub (limited current)
- Check USB cable for data capability (not charge-only)
- Look for boot log in serial monitor at 115200 baud (ESP32) or 74880 baud for boot messages (ESP8266)
- Check GPIO0 pin status (must be floating or HIGH for normal boot, LOW to enter flash mode)
- For battery-powered: measure voltage under WiFi transmit — should not drop below 3.0V

## Tools Needed
- Multimeter
- USB cable (data-capable)
- Arduino IDE or PlatformIO
- Logic level converter (if interfacing 5V signals)

## Step-by-Step Guidance
1. Open serial monitor at 115200 baud. Reset device. Read boot log for error messages.
2. If "Brownout detector triggered" → supply is too weak. Replace USB cable and/or use higher-current supply. Add 470µF capacitor across 3.3V and GND pins.
3. For flash upload failure → hold BOOT (GPIO0) button while clicking Upload. Release after "Connecting..." appears.
4. If serial not detected → install CP2102 or CH340 driver for your OS. Check Device Manager (Windows) or `ls /dev/tty*` (Mac/Linux) for port.
5. For WiFi dropping → check RSSI value: `WiFi.RSSI()`. Below -70dBm is marginal, below -80dBm is poor. Move device closer to router or add repeater.
6. For WiFi drops after hours of use → check available heap: `ESP.getFreeHeap()`. Below 10KB indicates memory leak in application. Restart device on schedule or fix leak.
7. For no networks found → test with minimal WiFi scan sketch. If still nothing → verify antenna (external antenna models: check IPEX connector is seated).
8. For periodic watchdog resets → add `yield()` or `delay(1)` inside long blocking loops in ESP8266. For ESP32, check task stack sizes.
9. For garbled serial on boot → first lines at 74880 baud are ESP8266 ROM boot message, then application takes over at your configured baud rate — this is normal.
10. For device crashing on battery → add 100µF–470µF ceramic or electrolytic capacitor between 3.3V and GND as close to chip as possible.

## Stop Conditions
- **STOP** if module is very hot — possible short circuit or reverse power
- **STOP** if connecting 5V directly to ESP32/ESP8266 GPIO — these are 3.3V logic, 5V damages them
- **STOP** if ESP32 is flashing in tight loop producing sparks — power off, inspect for solder short
- Do not leave GPIO0 permanently LOW after repair — device will always enter flash mode and not run application

## Prevention
- Always power ESP32/8266 from dedicated 3.3V regulator or LDO rated at 600mA+
- Add bulk capacitance (470µF) near power pins for WiFi peak current
- Use static IP assignment to avoid DHCP timeout WiFi drops
- Monitor heap memory in long-running applications and restart on low threshold
- Install serial drivers on all development machines before first use
